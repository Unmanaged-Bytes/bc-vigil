Overview

BitCrafts Vigil is a web frontend for bc-hash. Its goal: monitor file integrity on a server by periodically hashing file trees and comparing results to a reference baseline.

Any change (file modified, added, removed) raises an integrity event visible in the UI. A red badge in the nav next to Scans shows the number of unacknowledged drifts.

Getting started

  1. Create a target (a path to monitor) — Targets tab
  2. Run a first manual scan — it automatically becomes the baseline
  3. Add one or more schedules to automate scans
  4. Check results from the dashboard or Scans tab
  5. When drift is detected: analyze it, then either acknowledge or promote the scan as baseline

Targets

A target represents a path to scan. Fields:

  • Unique name
  • Absolute path (a ~ is expanded against the system user's home)
  • Algorithm: sha256 (robust, recommended), xxh128 / xxh3 (very fast, non-cryptographic), crc32 (light, for accidental corruption only)
  • Threads: auto, 0 (single-thread), or an integer

From a target page:

  • Duplicate — creates a copy (suffixed (copy)) with all parameters, handy for variants
  • Ack all — acknowledges all unacknowledged drifts for the target in one click
  • Delete — removes the target and all its scans, schedules, events

Include / Exclude

Filtering passed to bc-hash via --include / --exclude. Matching on basename only (no path prefix).

One pattern per line. Common exclusions:

  • .git — cuts out the entire git object store
  • node_modules — JS dependencies
  • *.log, *.tmp — volatile files
  • __pycache__, .venv — Python artefacts

Excluding a directory prunes its entire subtree before descent → big win on time and memory for large repos.

Baseline

  • The first successful scan of a target becomes the baseline automatically
  • Without a baseline, no drift is detected
  • Promote baseline button on a scan page (ok or drift) to freeze it as the new reference

A target's current baseline is protected from automatic purge.

Schedules

Background automatic trigger (APScheduler, UTC timezone). Six modes:

  • Every X minutes (1–59)
  • Every hour at a given minute
  • Every day at a given time
  • Every week on selected days
  • Every month on the Nth day
  • Cron expression (5 fields) for advanced cases

The preview shows the next 5 runs in real time during editing. A disabled schedule stays in DB but is not registered with the scheduler.

Schedule marked "stuck"

On a target page, each schedule shows the last actual run date. If no scheduled run has occurred for more than 2× the period (e.g. more than 2 days for a daily schedule), it is labeled stuck.

Typical causes: BitCrafts Vigil was stopped, bc-hash process failed repeatedly, or scans got stuck in running without cancellation. Check the Scans tab filtered on this target to diagnose.

Scans and events

Scan statuses:

  • pending — queued
  • running — bc-hash in progress
  • ok — hashed, no drift
  • drift — drift detected vs baseline
  • failed — error (binary, I/O, etc.)
  • cancelled — interrupted by user

Event types:

  • added — file present now, absent in baseline
  • removed — file present in baseline, absent now
  • modified — digest differs (content changed)

Peak RSS: bc-hash process peak RSS measured during execution (sampling /proc/<pid>/status every 50 ms). Useful to size the machine on large repos.

Filter and ack in bulk

The Scans tab offers:

  • Filters by status and target
  • Auto-refresh every 3 seconds when a scan is in progress
  • Ack all button: acknowledges all unacknowledged drifts (per-target option on the target page)

Acknowledge marks drift as "seen" — baseline stays intact, so a future scan will find the same drift. Promote changes the baseline.

CSV export

On a scan page with drift, Export CSV button → scan-<id>-events.csv with columns: 

scan_id,target,event_type,path,old_digest,new_digest

Handy for spreadsheet analysis or external SIEM integration.

Cancel a scan

On a running scan page:

  • Cancel — SIGTERM, scan marked cancelled, partial digest removed
  • Force stop — SIGKILL (when SIGTERM isn't enough)

Retention / auto-purge

Env var BC_VIGIL_SCAN_RETENTION_DAYS:

  • 0 (default) — no purge, everything kept
  • N > 0 — a 24h job deletes finished scans (and digests) whose started_at is older than now - N days

A target's current baseline and unfinished scans (pending / running) are always protected.

DB administration

Admin tab. Three actions:

  • Backup — downloads a .tar.gz (DB + digests). Can be taken while scans run.
  • Restore — upload a backup: replaces DB and digests. Automatic snapshot taken before.
  • Reset — wipes targets, schedules, scans and digests. Automatic snapshot taken before.

Blocked if scans are pending or running. Scheduler is stopped during the operation then restarted.

FAQ

Why does my first scan show no drift?
Expected behaviour: the first scan becomes the baseline, there's nothing to compare against yet.
Can a single file be monitored (not a folder)?
Yes — a target path can be a file or a directory.
What's the difference between ack and promote baseline?
Acknowledge: marks drift as "seen". Baseline stays the same, so a future scan will find the same drift again. Promote: adopts the current scan as the new reference, drift disappears for subsequent scans.
How do I delete old scans?
Either enable BC_VIGIL_SCAN_RETENTION_DAYS for auto-purge, delete a target (cascades all its scans), or do a full DB reset.
Which algorithm should I choose?
For malicious change detection: sha256. For accidental corruption on large volumes: xxh128 (much faster). crc32: only for light cases, more prone to collisions.
A schedule shows "stuck", what do I do?
Check that BitCrafts Vigil runs without interruption, look at recent scans of the target for successive failures or stuck scans. A restart will reset jobs.
Can I consume a scan's events from a script?
Yes, GET /scans/<id>/events.csv returns a CSV (Content-Type text/csv), usable via curl + jq/csvkit or in a SIEM.
