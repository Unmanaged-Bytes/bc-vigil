Overview

BitCrafts Vigil is a web frontend around two command-line tools:

  • bc-hash for the Integrity module: hashes a directory tree and compares with a baseline to detect drifts (added, removed, modified files).
  • bc-duplicate for the Duplicates module: finds identical files, lets you quarantine them and later restore or permanently purge.

Both modules share the same SQLite database, cron schedulers, backup / restore machinery and FR/EN UI. Two badges in the navbar count unacknowledged alerts (integrity drifts and dedup scans with duplicates).

Getting started

  1. Create an integrity target and/or a dedup target (Targets for integrity, Duplicates for dedup).
  2. Run a first manual scan: the integrity side adopts it as the baseline; the dedup side produces a report with duplicate groups.
  3. Optionally add one or more schedules to automate scans.
  4. Act on results: promote or acknowledge an integrity drift, quarantine selected duplicates.

Integrity module

Integrity targets

An integrity target represents a path to monitor.

  • Unique name.
  • Absolute path (~ resolves against the system user's home).
  • Algorithm: sha256 (robust, recommended for malicious-change detection), xxh128 / xxh3 (fast, non-cryptographic, recommended for large volumes / accidental corruption), crc32 (light, collision-prone, only for trivial cases).
  • Threads: auto, 0 (single-threaded) or an integer.

From the target page:

  • Duplicate: clones the target with the (copy) suffix.
  • Ack all: acknowledges every unacknowledged drift at once.
  • Delete: removes the target and all its scans, schedules, events.

Include / exclude

Filters passed to bc-hash as --include / --exclude. Matched on basename only (no path prefix). One pattern per line.

Common exclusions:

  • .git — prunes the git object store
  • node_modules — JS dependencies
  • *.log, *.tmp — volatile files
  • __pycache__, .venv — Python artefacts

Excluding a directory prunes the whole subtree before descent, which saves a lot of time and memory on large repos.

Baseline

  • The first successful scan becomes the baseline automatically.
  • Without a baseline, no scan reports any drift.
  • The Promote baseline button on any ok/drift scan page freezes that scan as the new reference.

The current baseline is protected from automatic retention purge.

Schedules

Background triggers (APScheduler, UTC timezone). Six modes:

  • Every X minutes (1–59)
  • Every hour at a specific minute
  • Daily at a specific time
  • Weekly on selected days
  • Monthly on the Nth day
  • 5-field cron expression for advanced cases

The preview shows the next 5 runs as you type. A disabled schedule is kept in DB but not registered with the scheduler.

Stuck schedule

On the target page, each schedule shows the last actual scheduled run. If no scheduled run has occurred for more than twice the period (e.g. more than 2 days for a daily schedule), it is labelled stuck. Typical causes: service was down, the binary kept failing, scans stayed in running without cancellation.

Integrity scans and events

Scan statuses:

  • pending — queued
  • running — executing
  • ok — hashed, no drift
  • drift — drift detected
  • failed — binary or I/O error
  • cancelled — interrupted

Event types:

  • added — present now, absent from baseline
  • removed — present in baseline, absent now
  • modified — digest changed

Peak RSS: resident-memory peak of the bc-hash process during the scan, sampled from /proc/<pid>/status.

Filter and ack in bulk

The Scans tab offers:

  • Status and target filters.
  • Automatic refresh every 3 seconds when a scan is active.
  • Ack all button, global or per target.

Acknowledge marks the drift as seen without touching the baseline, so a future scan will report the same drift. Promote adopts the scan as the new reference, the drift disappears for subsequent scans.

CSV export

On a scan-with-drift page, the Export CSV button downloads scan-<id>-events.csv: 

scan_id,target,event_type,path,old_digest,new_digest

Cancel a scan

  • Cancel: SIGTERM, scan marked cancelled, partial digest removed.
  • Force stop: SIGKILL, for cases where SIGTERM does not suffice.

Retention / automatic purge

BC_VIGIL_SCAN_RETENTION_DAYS:

  • 0 (default) — no purge.
  • N > 0 — a job runs every 24h and deletes finished scans (and their digests) older than now - N days.

Current baseline and non-finished scans are always protected.

Duplicates module

Overview

This module delegates duplicate discovery to bc-duplicate. It hashes in multiple phases (size filter, prefix/suffix fast hash, full hash on candidates) and returns a JSON report.

BitCrafts Vigil parses that report, persists duplicate groups in DB and offers two action paths:

  • Manual selection: tick files to quarantine, group by group.
  • Auto-resolve: apply a keep-rule to the whole scan in a single action.

Deletion is never immediate: selected files are first moved to a trash then purged after a configurable retention period.

Dedup targets

  • Unique name and absolute path.
  • Algorithm: xxh3 (default, fast), xxh128 (wider hash, very fast), sha256 (cryptographic, slower but collision-free).
  • Minimum size: skip files smaller than that (cuts the tiny-file noise).
  • Include / exclude: same conventions as for integrity (basename matching).
  • Options: include hidden files, follow symlinks, treat hardlinks as duplicates, do not cross filesystem boundaries.

Storage backend

On target creation, BitCrafts Vigil resolves the path to its mount point via /proc/self/mountinfo, reads /sys/block/<device>/queue/rotational, and classifies the backend:

  • SSD: no constraint, threads=auto by default.
  • HDD (rotational): threads is forced to 1. Multi-threaded reads on spinning rust can be 2 to 10 times slower than single-threaded. Tick "Force threads value" to keep your input anyway (useful for very large RAM caches, SSD caches, etc.).
  • Removable (USB, etc.): only a label on the target page.
  • Network filesystem (NFS, SMB/CIFS, SSHFS, 9p, Ceph, AFS, GlusterFS, rclone via FUSE): target creation is refused unless "Accept network filesystem" is ticked. Performance is not guaranteed over network FS; the opt-in is an explicit reminder.
  • Unknown: displayed as-is, no constraint.

The detected backend is shown on each target page with the block device name (nvme0n1, sda, …).

Dedup schedules

Dedup schedules work exactly like integrity ones (same modes, same UI, same UTC timezone). If a scheduled run fires while a scan is already active for the same target, it is simply skipped (see Per-target lock).

Dedup scans

Dedup scan statuses:

  • pending — queued
  • running — executing
  • ok — no duplicates
  • duplicates — duplicates found
  • failed — error
  • cancelled — interrupted

Acknowledging a duplicates scan does not touch any file: it only clears the navbar badge. Useful once the duplicates have been handled or explicitly accepted as legitimate.

The scan detail page is paginated: 20 groups per page, sorted by size descending by default (so you focus on the biggest wins first). Sort toggle and page navigation are provided. Above 10,000 groups, a banner explicitly recommends using auto-resolve rather than manual ticking.

Auto-resolve

Four keep-rules. Each picks one survivor per group and proposes to quarantine the rest:

  • Shortest path: survivor = lexicographically shortest path (handy when copies live deeper than the original).
  • Oldest (mtime): survivor = file with the oldest modification timestamp.
  • Newest (mtime): survivor = file with the newest modification timestamp.
  • Priority folder: survivor = file under the given prefix (Priority folder field). Groups with no file under that prefix are skipped.

The rule is applied to all groups in the scan, not only the visible page. The result opens the preview page which you still have to confirm with DELETE — nothing moves until that confirmation.

Manual selection

On the scan detail page, each group lists its files with a checkbox. Three shortcuts to avoid clicking one-by-one:

  • all but first button on a specific group.
  • Tick all but first (visible page) at the top of the form (applies to the current page only).
  • Clear all.

Safety rule: at least one survivor per group. Any attempt to tick every file of a group is rejected in the preview.

Preview and confirm

Before any action, the preview shows:

  • Number of files to move, total size, destination.
  • Skipped files with the reason (gone, became symlink, size drifted, hash no longer matches survivor, survivor missing, etc.).
  • Cross-FS warning if some files are not on the same filesystem as the trash (copy_unlink mode, slower than rename).
  • Bulk-threshold warning if the selection exceeds BC_VIGIL_DEDUP_DELETION_BULK_THRESHOLD (500 by default): an explicit "I confirm" box must be ticked.

Confirmation requires typing DELETE. At that point:

  1. A SHA-256 TOCTOU check is redone at move-time, files modified since the preview are rejected.
  2. Files are moved (rename atomic if possible, else copy2 + fsync + unlink).
  3. A DedupDeletion row is persisted with origin, destination, hash and caller.

Trash

Quarantined files live under BC_VIGIL_DEDUP_TRASH_DIR (default <data_dir>/dedup/trash/, mode 0700, owner bc-vigil). Each item lives in scan-<id>/group-<id>/<uuid>/ with a meta.json describing the original.

The Trash page lists all quarantined items with:

  • quarantined — awaiting, restorable.
  • restored — returned to original location.
  • purged — permanently deleted.
  • failed — a step failed (audit trail kept).

Two actions on quarantined items:

  • Restore: move the file back. Refused if another file now sits at the original path.
  • Purge: immediate, irreversible delete.

An APScheduler job runs every 24h and automatically purges items older than BC_VIGIL_DEDUP_TRASH_RETENTION_DAYS (7 days by default). Set to 0 to disable automatic purge.

Per-target lock

Only one scan may be active (pending/running) per target at a time. A new manual trigger silently redirects to the active scan; a scheduled run that tries to fire while another scan is active is skipped (a log line is emitted).

The lock is stored in the DB, so it survives restarts. If a scan remains stuck in running after a crash, use Force stop from its detail page to free it.

Administration

Backup / Restore / Reset

Admin tab. Three actions:

  • Backup: download a .tar.gz with the SQLite DB, integrity digests and dedup JSON reports. A PRAGMA wal_checkpoint(TRUNCATE) is forced first so the archived DB file is self-contained.
  • Restore: upload a previously produced archive. Replaces DB + digests + dedup directory + trash. An automatic snapshot is taken first.
  • Reset: wipe the DB and all related files (digests, dedup reports, trash). An automatic snapshot is also taken first.

The last two are blocked as long as an integrity or dedup scan is pending/running. Schedulers are stopped during the operation and restarted afterwards.

Concurrency (WAL)

The SQLite DB is opened with journal_mode=WAL, synchronous=NORMAL, busy_timeout=5000. This lets UI reads stay snappy while large writes are flushed (duplicate groups persistence at the end of a big scan).

The bc-vigil.sqlite-wal and bc-vigil.sqlite-shm files are created automatically by SQLite — this is normal and expected in WAL mode. Backup calls an explicit checkpoint so only the main file is archived.

Storage module

The Storage tab shows per-mount disk usage: total, used, free, filesystem type. Informational only, no impact on scans.

FAQ

Why does my first integrity scan report no drift?
Expected: the first scan becomes the baseline, there is nothing to compare against.
Acknowledge vs promote baseline?
Acknowledge marks the drift as seen without touching the baseline — a future scan will report the same drift. Promote adopts the scan as the new reference and the drift disappears for subsequent scans.
Does acknowledging a dedup scan delete any file?
No. Acknowledging is only a visual marker that clears the navbar badge. To actually free space you must go through quarantine then purge (manual or automatic).
Why is threads forced to 1 on my target?
BitCrafts Vigil detected a rotational disk via /sys/block/<dev>/queue/rotational. On HDD, parallel reads trigger head thrashing and can be 2–10× slower than single-threaded. Tick "Force threads value" at creation to keep your value anyway.
Why was target creation refused on my NFS share?
Network filesystems (NFS, SMB, SSHFS, 9p, Ceph, AFS, GlusterFS, rclone) cannot guarantee performance. Tick "Accept network filesystem" to bypass the refusal.
I triggered a scan but nothing happens — another scan is active.
The per-target lock blocks a second simultaneous scan. The redirect link points to the active scan. To free a stuck scan, use Force stop from its detail page.
Can I undo a quarantine?
Yes, as long as the file has not been purged: click Restore in the trash. Restoration fails if another file has since taken the original path.
My trash fills /var/lib, what now?
Point BC_VIGIL_DEDUP_TRASH_DIR to a larger disk (same FS as the scanned targets ideally, so rename stays atomic).
A scan seems slower via the UI than from the command line.
The UI adds the JSON report parsing and group persistence. The overhead is proportional to the group count. Since 0.5.0 the persistence is chunked to keep the UI responsive during the final flush.
How do I consume a scan's groups from a script?
GET /dedup/scans/<id>/groups.csv returns a CSV (scan_id,target,group_id,size,path) usable with curl + jq/csvkit.
A schedule is "stuck", what should I do?
Check that BitCrafts Vigil runs continuously, inspect recent scans of the target to spot failures or stuck scans. Restarting the service re-registers the jobs.
