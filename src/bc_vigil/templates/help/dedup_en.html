Duplicates module

Overview

This module delegates duplicate discovery to bc-duplicate. It hashes in multiple phases (size filter, prefix/suffix fast hash, full hash on candidates) and returns a JSON report.

BitCrafts Vigil parses that report, persists duplicate groups in DB and offers two action paths:

  • Manual selection: tick files to quarantine, group by group.
  • Auto-resolve: apply a keep-rule to the whole scan in a single action.

Deletion is never immediate: selected files are first moved to a trash then purged after a configurable retention period.

Dedup targets

  • Unique name and absolute path.
  • Algorithm: xxh3 (default, fast), xxh128 (wider hash, very fast), sha256 (cryptographic, slower but collision-free).
  • Minimum size: skip files smaller than that (cuts the tiny-file noise).
  • Include / exclude: same conventions as for integrity (basename matching).
  • Options: include hidden files, follow symlinks, treat hardlinks as duplicates, do not cross filesystem boundaries.

Storage backend

On target creation, BitCrafts Vigil resolves the path to its mount point via /proc/self/mountinfo, reads /sys/block/<device>/queue/rotational, and classifies the backend:

  • SSD: no constraint, threads=auto by default.
  • HDD (rotational): threads is forced to 1. Multi-threaded reads on spinning rust can be 2 to 10 times slower than single-threaded. Tick "Force threads value" to keep your input anyway (useful for very large RAM caches, SSD caches, etc.).
  • Removable (USB, etc.): only a label on the target page.
  • Network filesystem (NFS, SMB/CIFS, SSHFS, 9p, Ceph, AFS, GlusterFS, rclone via FUSE): target creation is refused unless "Accept network filesystem" is ticked. Performance is not guaranteed over network FS; the opt-in is an explicit reminder.
  • Unknown: displayed as-is, no constraint.

The detected backend is shown on each target page with the block device name (nvme0n1, sda, …).

Dedup schedules

Dedup schedules work exactly like integrity ones (same modes, same UI, same UTC timezone). If a scheduled run fires while a scan is already active for the same target, it is simply skipped (see Per-target lock).

Dedup scans

Dedup scan statuses:

  • pending — queued
  • running — executing
  • ok — no duplicates
  • duplicates — duplicates found
  • failed — error
  • cancelled — interrupted

Acknowledging a duplicates scan does not touch any file: it only clears the navbar badge. Useful once the duplicates have been handled or explicitly accepted as legitimate.

The scan detail page is paginated: 20 groups per page, sorted by size descending by default (so you focus on the biggest wins first). Sort toggle and page navigation are provided. Above 10,000 groups, a banner explicitly recommends using auto-resolve rather than manual ticking.

Auto-resolve

Four keep-rules. Each picks one survivor per group and proposes to quarantine the rest:

  • Shortest path: survivor = lexicographically shortest path (handy when copies live deeper than the original).
  • Oldest (mtime): survivor = file with the oldest modification timestamp.
  • Newest (mtime): survivor = file with the newest modification timestamp.
  • Priority folder: survivor = file under the given prefix (Priority folder field). Groups with no file under that prefix are skipped.

The rule is applied to all groups in the scan, not only the visible page. The result opens the preview page which you still have to confirm with DELETE — nothing moves until that confirmation.

Manual selection

On the scan detail page, each group lists its files with a checkbox. Three shortcuts to avoid clicking one-by-one:

  • all but first button on a specific group.
  • Tick all but first (visible page) at the top of the form (applies to the current page only).
  • Clear all.

Safety rule: at least one survivor per group. Any attempt to tick every file of a group is rejected in the preview.

Preview and confirm

Before any action, the preview shows:

  • Number of files to move, total size, destination.
  • Skipped files with the reason (gone, became symlink, size drifted, hash no longer matches survivor, survivor missing, etc.).
  • Cross-FS warning if some files are not on the same filesystem as the trash (copy_unlink mode, slower than rename).
  • Bulk-threshold warning if the selection exceeds BC_VIGIL_DEDUP_DELETION_BULK_THRESHOLD (500 by default): an explicit "I confirm" box must be ticked.

Confirmation requires typing DELETE. At that point:

  1. A SHA-256 TOCTOU check is redone at move-time, files modified since the preview are rejected.
  2. Files are moved (rename atomic if possible, else copy2 + fsync + unlink).
  3. A DedupDeletion row is persisted with origin, destination, hash and caller.

Trash

Quarantined files live under BC_VIGIL_DEDUP_TRASH_DIR (default <data_dir>/dedup/trash/, mode 0700, owner bc-vigil). Each item lives in scan-<id>/group-<id>/<uuid>/ with a meta.json describing the original.

The Trash page lists all quarantined items with:

  • quarantined — awaiting, restorable.
  • restored — returned to original location.
  • purged — permanently deleted.
  • failed — a step failed (audit trail kept).

Two actions on quarantined items:

  • Restore: move the file back. Refused if another file now sits at the original path.
  • Purge: immediate, irreversible delete.

An APScheduler job runs every 24h and automatically purges items older than BC_VIGIL_DEDUP_TRASH_RETENTION_DAYS (7 days by default). Set to 0 to disable automatic purge.

Per-target lock

Only one scan may be active (pending/running) per target at a time. A new manual trigger silently redirects to the active scan; a scheduled run that tries to fire while another scan is active is skipped (a log line is emitted).

The lock is stored in the DB, so it survives restarts. If a scan remains stuck in running after a crash, use Force stop from its detail page to free it.
