Administration

Sauvegarde / Restauration / Réinitialisation

Onglet Admin. Trois actions :

  • Sauvegarde : télécharge un .tar.gz contenant la DB SQLite, les digests d'intégrité et les rapports JSON des scans doublons. Un PRAGMA wal_checkpoint(TRUNCATE) est forcé avant l'archivage pour garantir que le fichier DB est complet.
  • Restauration : upload d'une archive précédemment produite. Remplace DB + digests + répertoire doublons + corbeille. Un snapshot automatique de l'état courant est pris avant.
  • Réinitialisation : efface toute la base et tous les fichiers liés (digests, rapports doublons, corbeille). Un snapshot automatique est également pris avant.

Les deux dernières opérations sont bloquées tant qu'un scan intégrité ou doublons est encore pending ou running. Les schedulers sont arrêtés le temps de l'opération puis redémarrés automatiquement.

Concurrence (WAL)

La base SQLite est ouverte en mode journal_mode=WAL, synchronous=NORMAL, busy_timeout=5000. Cela permet aux lectures de l'UI de rester rapides pendant les écritures lourdes (persistance des groupes à la fin d'un gros scan).

Les fichiers bc-vigil.sqlite-wal et bc-vigil.sqlite-shm sont créés automatiquement par SQLite — c'est normal et attendu en mode WAL. Le backup appelle un checkpoint explicite pour que l'archive ne contienne que le fichier principal.

Observabilité

Deux endpoints non authentifiés, pensés pour un monitoring externe (Prometheus, healthcheck Docker, curl en cron) :

  • GET /health — JSON avec status, version, db, scheduler_integrity, scheduler_dedup. Renvoie 200 si tout est vert, 503 si la DB est injoignable ou qu'un scheduler est arrêté.
  • GET /metrics — format exposition Prometheus. Expose bc_vigil_up, bc_vigil_info{version}, bc_vigil_db_up, bc_vigil_scheduler_up{module}, bc_vigil_scans_total{module, status}, bc_vigil_dedup_deletions_total{status}. Intervalle de scrape recommandé : 30 s.

Au démarrage du service, les scans laissés en pending ou running par un précédent process tué sont automatiquement marqués failed avec le message « interrompu par un redémarrage du service ». Une ligne INFO dans les logs indique le nombre de scans nettoyés. Un job de VACUUM SQLite tourne le 1er de chaque mois à 04:00 (fuseau local) pour récupérer l'espace après les purges de rétention.

Stockage (module)

L'onglet Stockage présente l'utilisation disque par point de montage : total, utilisé, libre, type de FS. C'est purement informatif, aucun impact sur les scans.
