Module Doublons

Vue d'ensemble

Ce module délègue la découverte de fichiers identiques à bc-duplicate. Il hache en plusieurs phases (filtre par taille, hash de prefixe/suffixe rapide, hash complet sur les candidats) puis retourne un rapport JSON.

BitCrafts Vigil parse ce rapport, persiste les groupes en base et offre deux chemins pour agir :

  • Sélection manuelle : cocher les fichiers à quarantainer, groupe par groupe.
  • Résolution automatique : appliquer une règle de conservation à tout le scan en une action.

La suppression n'est jamais immédiate : les fichiers sélectionnés sont d'abord déplacés dans une corbeille puis purgés après une période de rétention configurable.

Cibles de recherche de doublons

  • Nom unique et chemin absolu.
  • Algorithme : xxh3 (défaut, rapide), xxh128 (plus large, très rapide), sha256 (cryptographique, plus lent mais immunisé aux collisions adverses).
  • Taille minimale : ignore les fichiers plus petits (pratique pour éviter le bruit des tout petits fichiers).
  • Inclure / exclure : mêmes conventions que pour l'intégrité (matching basename).
  • Options : inclure fichiers cachés, suivre les liens symboliques, considérer les liens physiques comme doublons, ne pas traverser les points de montage.

Support de stockage

À la création d'une cible, BitCrafts Vigil résout le chemin vers son point de montage via /proc/self/mountinfo, lit /sys/block/<device>/queue/rotational puis classe le support dans l'une des catégories :

  • SSD : aucune contrainte, threads=auto par défaut.
  • HDD (disque mécanique) : threads est forcé à 1, car des lectures multi-thread sur un disque mécanique peuvent être 2 à 10 fois plus lentes que du mono-thread. La case « Forcer la valeur de threads » permet tout de même de conserver la valeur saisie pour les cas particuliers (très gros cache RAM, SSD en cache, etc.).
  • Amovible (clé USB, etc.) : rien de spécial, juste un étiquetage dans la fiche.
  • Système de fichiers réseau (NFS, SMB/CIFS, SSHFS, 9p, Ceph, AFS, GlusterFS, rclone via FUSE) : la création est refusée sauf si la case « J'accepte le système de fichiers réseau » est cochée. La performance n'est pas garantie sur les FS réseau — l'opt-in est un rappel explicite.
  • Inconnu : aucune contrainte, juste affiché pour information.

Le support détecté est affiché sur la fiche de chaque cible, avec le nom du block device (nvme0n1, sda, etc.).

Schedules doublons

Les schedules du module doublons fonctionnent exactement comme ceux de l'intégrité (mêmes modes, même UI, même fuseau UTC). Un schedule planifié qui démarre alors qu'un scan est déjà actif pour la même cible est simplement ignoré (voir Lock par cible).

Analyses doublons

Chaque scan doublons passe par ces statuts :

  • pending — en file d'attente
  • running — exécution en cours
  • ok — aucun doublon trouvé
  • duplicates — doublons trouvés
  • failed — erreur
  • cancelled — interrompu

Acquitter un scan duplicates ne touche aucun fichier : c'est uniquement un marqueur visuel (le badge Analyses doublons dans la barre de navigation disparaît pour ce scan). Utile après avoir traité les doublons ou pour accepter que le doublon est légitime.

La page de détail d'un scan est paginée à 20 groupes par page, triés par taille décroissante par défaut (on cible les plus gros gaspillages en premier). Filtre de tri, navigation entre pages. Au-delà de 10 000 groupes, un bandeau recommande explicitement la résolution automatique plutôt que le clic à clic.

Résolution automatique

Quatre règles de conservation. Chacune garde un survivant par groupe et propose de quarantainer le reste :

  • Chemin le plus court : survivant = chemin lexicographiquement le plus court (utile quand les copies sont plus profondes dans l'arborescence que l'original).
  • Plus ancien (mtime) : survivant = fichier avec la date de modification la plus ancienne.
  • Plus récent (mtime) : survivant = fichier avec la date de modification la plus récente.
  • Dossier prioritaire : survivant = fichier situé sous le préfixe indiqué (champ Dossier prioritaire). Si un groupe n'a aucun fichier sous ce préfixe, il est ignoré par la règle.

La règle s'applique à tous les groupes du scan, pas seulement la page visible. Le résultat ouvre la page d'aperçu que vous devez confirmer avec DELETE — rien n'est déplacé tant que vous n'avez pas confirmé.

Sélection manuelle

Sur la page de détail d'un scan, chaque groupe liste ses fichiers avec une case à cocher. Trois raccourcis pour éviter le clic à clic :

  • Bouton tout sauf le premier sur un groupe particulier (coche tous les fichiers sauf le premier du groupe).
  • Bouton Cocher tout sauf le premier (page affichée) en haut du formulaire (toute la page courante).
  • Bouton Tout décocher.

La règle de sûreté : il doit rester au moins un survivant par groupe. Toute tentative de cocher tous les fichiers d'un groupe est rejetée dans l'aperçu.

Aperçu et confirmation

L'aperçu montre avant toute action :

  • Nombre de fichiers à déplacer, taille totale, destination.
  • Fichiers ignorés (avec la raison : disparu, devenu lien, taille modifiée, hash différent du survivant, survivant absent, etc.).
  • Avertissement cross-FS si certains fichiers ne sont pas sur le même système de fichiers que la corbeille (mode copy_unlink plus lent que rename).
  • Avertissement seuil bulk si la sélection dépasse BC_VIGIL_DEDUP_DELETION_BULK_THRESHOLD (500 par défaut) : une case « Je confirme » doit être cochée.

La confirmation exige de taper DELETE dans le champ dédié. À ce moment :

  1. Un contrôle SHA-256 de TOCTOU est refait au moment du déplacement pour rejeter les fichiers modifiés depuis l'aperçu.
  2. Les fichiers sont déplacés (rename atomique si possible, sinon copy2 + fsync + unlink).
  3. Une entrée DedupDeletion est créée en base avec l'origine, la destination dans la corbeille, le hash et l'auteur.

Corbeille

Les fichiers mis en quarantaine vivent sous le chemin défini par BC_VIGIL_DEDUP_TRASH_DIR (par défaut <data_dir>/dedup/trash/, mode 0700, propriétaire bc-vigil). Chaque élément est rangé dans scan-<id>/group-<id>/<uuid>/ avec un meta.json à côté qui décrit l'origine exacte.

La page Corbeille liste tous les éléments en quarantaine avec leur statut :

  • quarantined — en attente, restaurable.
  • restored — restauré à son emplacement d'origine.
  • purged — supprimé définitivement.
  • failed — une étape a échoué (audit trail conservé).

Deux actions disponibles sur un élément en quarantaine :

  • Restaurer : ramène le fichier à son chemin d'origine. Refusé si un autre fichier a pris la place entre-temps.
  • Purger : suppression immédiate et irréversible.

Un job APScheduler tourne toutes les 24 h et purge automatiquement les éléments plus anciens que BC_VIGIL_DEDUP_TRASH_RETENTION_DAYS (7 jours par défaut). Mettre à 0 désactive la purge automatique.

Lock par cible

Un seul scan peut être actif (pending ou running) par cible à la fois. Un nouveau déclenchement manuel redirige silencieusement vers le scan actif ; un schedule qui essaie de partir alors qu'un scan est déjà en cours est ignoré (une ligne de log le signale).

Le verrou est porté en base de données : il survit au redémarrage du service. Si un scan reste bloqué en running à la suite d'un crash, utilisez Forcer l'arrêt sur la fiche du scan pour le libérer.
