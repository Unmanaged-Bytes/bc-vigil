Module Intégrité

Cibles intégrité

Une cible intégrité représente un chemin à surveiller.

  • Nom unique
  • Chemin absolu (un ~ est résolu contre le home système)
  • Algorithme : sha256 (robuste, recommandé pour détection malveillante), xxh128 / xxh3 (rapides, non cryptographiques, recommandés pour gros volumes et corruption accidentelle), crc32 (léger, collisions plus probables, à réserver aux cas simples)
  • Threads : auto, 0 (mono-thread) ou un entier

Depuis la fiche d'une cible :

  • Modifier : édite le nom, l'algorithme, les threads et les patterns include/exclude sans perdre la baseline ni l'historique des scans. Le chemin est immuable (une cible = un chemin).
  • Dupliquer : crée une copie suffixée (copie).
  • Tout acquitter : acquitte en un clic toutes les dérives non acquittées.
  • Supprimer : efface la cible et tous ses scans, schedules et événements.

Include / exclude

Filtrage passé à bc-hash via --include / --exclude. Matching sur basename uniquement (pas de préfixe de chemin). Un pattern par ligne.

Exclusions typiques :

  • .git — coupe le store d'objets des dépôts git
  • node_modules — dépendances JS
  • *.log, *.tmp — fichiers volatiles
  • __pycache__, .venv — artefacts Python

Exclure un répertoire prune tout son sous-arbre avant descente, ce qui fait gagner beaucoup de temps et de mémoire sur les gros dépôts.

Baseline

  • Le premier scan réussi devient automatiquement la baseline.
  • Tant qu'il n'y a pas de baseline, un scan ne détecte aucune dérive.
  • Le bouton Promouvoir baseline (fiche d'un scan ok ou drift) fige le scan comme nouvelle référence.

La baseline courante d'une cible est protégée contre la purge automatique.

Schedules

Déclenchement automatique en arrière-plan (APScheduler, fuseau UTC). Six modes :

  • Toutes les X minutes (1–59)
  • Chaque heure à une minute donnée
  • Chaque jour à une heure donnée
  • Chaque semaine sur une sélection de jours
  • Chaque mois le Ne jour
  • Expression cron à 5 champs pour les cas avancés

L'aperçu affiche les 5 prochaines exécutions pendant la saisie. Un schedule désactivé reste en base mais n'est pas enregistré auprès du scheduler. Le bouton Modifier sur la fiche d'une cible permet de changer le cron et l'état enabled sans supprimer le schedule (le formulaire s'ouvre en mode expression cron pré-remplie ; basculer vers les modes quotidien / hebdo / mensuel depuis la liste de fréquence).

Schedule bloqué

Sur la fiche d'une cible, chaque schedule affiche la date du dernier run réel. Si aucun run programmé n'a eu lieu depuis plus de deux fois la période (par exemple, plus de 2 jours pour un schedule quotidien), il est étiqueté stuck. Causes typiques : service à l'arrêt, échecs successifs du binaire, scans restés en running sans annulation.

Scans intégrité et événements

Chaque scan passe par ces statuts :

  • pending — en file d'attente
  • running — exécution en cours
  • ok — hashé, aucune dérive
  • drift — dérive détectée
  • failed — erreur binaire ou I/O
  • cancelled — interrompu

Types d'événements :

  • added — fichier présent maintenant, absent de la baseline
  • removed — présent dans la baseline, absent maintenant
  • modified — digest différent

Peak RSS : pic de mémoire résidente du process bc-hash pendant le scan. Échantillonné via /proc/<pid>/status.

Filtrer et acquitter en masse

L'onglet Scans propose :

  • Filtres par statut et par cible.
  • Actualisation automatique toutes les 3 secondes lorsqu'un scan est en cours.
  • Bouton Tout acquitter, global ou par cible.

Acquitter marque la dérive comme vue sans toucher à la baseline : un futur scan retrouvera la même dérive. Promouvoir adopte le scan comme nouvelle référence, la dérive disparaît pour les scans suivants.

Export CSV

Sur la fiche d'un scan avec dérive, le bouton Export CSV télécharge scan-<id>-events.csv au format : 

scan_id,target,event_type,path,old_digest,new_digest

Annuler un scan

  • Annuler : SIGTERM, scan marqué cancelled, digest partiel supprimé.
  • Forcer l'arrêt : SIGKILL, pour les cas où SIGTERM ne suffit pas.

Rétention / purge automatique

Variable BC_VIGIL_SCAN_RETENTION_DAYS :

  • 0 (défaut) — aucune purge.
  • N > 0 — un job tourne toutes les 24 h et supprime les scans terminés (et leurs digests) plus vieux que now - N jours.

La baseline courante et les scans non terminés sont toujours protégés.
