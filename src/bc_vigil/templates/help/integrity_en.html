Integrity module

Integrity targets

An integrity target represents a path to monitor.

  • Unique name.
  • Absolute path (~ resolves against the system user's home).
  • Algorithm: sha256 (robust, recommended for malicious-change detection), xxh128 / xxh3 (fast, non-cryptographic, recommended for large volumes / accidental corruption), crc32 (light, collision-prone, only for trivial cases).
  • Threads: auto, 0 (single-threaded) or an integer.

From the target page:

  • Edit: change the name, algorithm, threads and include/exclude patterns without losing the baseline or the scan history. The path is immutable (a target is defined by its path).
  • Duplicate: clones the target with the (copy) suffix.
  • Ack all: acknowledges every unacknowledged drift at once.
  • Delete: removes the target and all its scans, schedules, events.

Include / exclude

Filters passed to bc-hash as --include / --exclude. Matched on basename only (no path prefix). One pattern per line.

Common exclusions:

  • .git — prunes the git object store
  • node_modules — JS dependencies
  • *.log, *.tmp — volatile files
  • __pycache__, .venv — Python artefacts

Excluding a directory prunes the whole subtree before descent, which saves a lot of time and memory on large repos.

Baseline

  • The first successful scan becomes the baseline automatically.
  • Without a baseline, no scan reports any drift.
  • The Promote baseline button on any ok/drift scan page freezes that scan as the new reference.

The current baseline is protected from automatic retention purge.

Schedules

Background triggers (APScheduler, UTC timezone). Six modes:

  • Every X minutes (1–59)
  • Every hour at a specific minute
  • Daily at a specific time
  • Weekly on selected days
  • Monthly on the Nth day
  • 5-field cron expression for advanced cases

The preview shows the next 5 runs as you type. A disabled schedule is kept in DB but not registered with the scheduler. The Edit button on the schedule row lets you change the cron expression and the enabled flag without dropping the schedule row (the form opens in pre-filled cron-expression mode; switch to daily / weekly / monthly through the frequency dropdown).

Stuck schedule

On the target page, each schedule shows the last actual scheduled run. If no scheduled run has occurred for more than twice the period (e.g. more than 2 days for a daily schedule), it is labelled stuck. Typical causes: service was down, the binary kept failing, scans stayed in running without cancellation.

Integrity scans and events

Scan statuses:

  • pending — queued
  • running — executing
  • ok — hashed, no drift
  • drift — drift detected
  • failed — binary or I/O error
  • cancelled — interrupted

Event types:

  • added — present now, absent from baseline
  • removed — present in baseline, absent now
  • modified — digest changed

Peak RSS: resident-memory peak of the bc-hash process during the scan, sampled from /proc/<pid>/status.

Filter and ack in bulk

The Scans tab offers:

  • Status and target filters.
  • Automatic refresh every 3 seconds when a scan is active.
  • Ack all button, global or per target.

Acknowledge marks the drift as seen without touching the baseline, so a future scan will report the same drift. Promote adopts the scan as the new reference, the drift disappears for subsequent scans.

CSV export

On a scan-with-drift page, the Export CSV button downloads scan-<id>-events.csv: 

scan_id,target,event_type,path,old_digest,new_digest

Cancel a scan

  • Cancel: SIGTERM, scan marked cancelled, partial digest removed.
  • Force stop: SIGKILL, for cases where SIGTERM does not suffice.

Retention / automatic purge

BC_VIGIL_SCAN_RETENTION_DAYS:

  • 0 (default) — no purge.
  • N > 0 — a job runs every 24h and deletes finished scans (and their digests) older than now - N days.

Current baseline and non-finished scans are always protected.
