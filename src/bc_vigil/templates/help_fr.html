Présentation

BitCrafts Vigil est une interface web pour bc-hash. Son objectif : surveiller l'intégrité de fichiers sur un serveur en hachant périodiquement des arborescences puis en comparant le résultat à une baseline de référence.

Un changement (fichier modifié, ajouté, supprimé) déclenche un événement d'intégrité visible dans l'UI. Un badge rouge dans la nav à côté de Scans indique le nombre de dérives non acquittées.

Prise en main

  1. Créer un target (un chemin à surveiller) — onglet Targets
  2. Lancer un premier scan manuel — il devient automatiquement la baseline
  3. Ajouter un ou plusieurs schedules pour automatiser les scans
  4. Consulter les résultats depuis le dashboard ou l'onglet Scans
  5. Quand une dérive est détectée : l'analyser, puis soit acquitter, soit promouvoir le scan en baseline

Targets

Un target représente un chemin à scanner. Champs :

  • Nom unique
  • Chemin absolu (un ~ est résolu contre le home de l'utilisateur système)
  • Algorithme : sha256 (robuste, recommandé), xxh128 / xxh3 (très rapides, non cryptographiques), crc32 (léger, pour détecter des corruptions accidentelles seulement)
  • Threads : auto, 0 (mono-thread), ou un entier

Depuis la fiche d'un target :

  • Dupliquer — crée une copie (suffixée (copie)) avec tous les paramètres, pratique pour créer des variantes
  • Tout acquitter — acquitte en un clic toutes les dérives non acquittées du target
  • Supprimer — efface le target et tous ses scans, schedules, événements

Include / Exclude

Filtrage passé à bc-hash via --include / --exclude. Matching sur basename uniquement (pas de chemin préfixe).

Un pattern par ligne. Exemples fréquents à exclure :

  • .git — coupe tout le store d'objets d'un dépôt git
  • node_modules — dépendances JS
  • *.log, *.tmp — fichiers volatiles
  • __pycache__, .venv — artefacts Python

Exclure un répertoire prune tout son sous-arbre avant descente → gros gain de temps et de mémoire sur de gros repos.

Baseline

  • Le premier scan réussi d'un target devient automatiquement la baseline
  • Tant qu'il n'y a pas de baseline, un scan ne détecte aucune dérive
  • Bouton Promouvoir baseline sur la fiche d'un scan (ok ou drift) pour le figer comme nouvelle référence

La baseline courante d'un target est protégée contre la purge automatique.

Schedules

Déclenchement automatique en arrière-plan (APScheduler, fuseau UTC). Six modes de configuration :

  • Toutes les X minutes (1–59)
  • Chaque heure à une minute donnée
  • Chaque jour à une heure donnée
  • Chaque semaine sur une sélection de jours
  • Chaque mois le Nième jour
  • Expression cron (5 champs) pour les cas avancés

L'aperçu affiche les 5 prochaines exécutions en temps réel pendant la saisie. Un schedule désactivé reste en base mais n'est pas enregistré auprès du scheduler.

Schedule marqué "stuck"

Sur la fiche d'un target, chaque schedule affiche la date du dernier run réel. Si aucun run programmé n'a eu lieu depuis plus de 2x la période (ex : plus de 2 jours pour un schedule quotidien), le schedule est étiqueté stuck.

Causes typiques : BitCrafts Vigil était à l'arrêt, le process bc-hash a échoué de manière répétée, ou les scans sont restés bloqués en running sans annulation. Consulter l'onglet Scans filtré sur ce target pour diagnostiquer.

Scans et événements

Chaque scan passe par ces statuts :

  • pending — en file
  • running — bc-hash en cours
  • ok — hashé, aucune dérive
  • drift — dérive détectée par rapport à la baseline
  • failed — erreur (binaire, I/O, etc.)
  • cancelled — interrompu par l'utilisateur

Types d'événements :

  • added — fichier présent maintenant, absent dans la baseline
  • removed — fichier présent dans la baseline, absent maintenant
  • modified — digest différent (contenu changé)

Peak RSS : pic de RSS du process bc-hash pendant l'exécution (échantillonnage de /proc/<pid>/status toutes les 50 ms). Utile pour dimensionner la machine sur de gros repos.

Filtrer et acquitter en masse

L'onglet Scans propose :

  • Filtres par statut et par target
  • Actualisation automatique toutes les 3 secondes lorsqu'un scan est en cours
  • Bouton Tout acquitter : acquitte toutes les dérives non acquittées (option par-target depuis la fiche du target)

Acquitter marque la dérive comme "vue" — la baseline reste intacte, donc un futur scan retrouvera la même dérive. Promouvoir change la baseline.

Export CSV

Sur la fiche d'un scan avec dérive, bouton Export CSV → scan-<id>-events.csv contenant : 

scan_id,target,event_type,path,old_digest,new_digest

Utile pour analyse dans un tableur ou intégration dans un SIEM externe.

Annuler un scan

Sur la fiche d'un scan en cours :

  • Annuler — SIGTERM, scan marqué cancelled, digest partiel supprimé
  • Forcer l'arrêt — SIGKILL (pour les cas où SIGTERM ne suffit pas)

Rétention / purge automatique

Variable d'environnement BC_VIGIL_SCAN_RETENTION_DAYS :

  • 0 (défaut) — aucune purge, tout est conservé
  • N > 0 — un job tourne toutes les 24 h et supprime les scans terminés (et leurs digests) dont started_at est antérieur à now - N jours

La baseline courante d'un target et les scans non terminés (pending / running) sont toujours protégés.

Administration DB

Onglet Admin. Trois actions :

  • Backup — téléchargement d'un .tar.gz (DB + digests). Peut être pris à chaud.
  • Restaurer — upload d'un backup : remplace DB et digests. Snapshot automatique pris avant.
  • Réinitialiser — efface targets, schedules, scans et digests. Snapshot automatique pris avant.

Bloqué si des scans sont pending ou running. Le scheduler est arrêté pendant l'opération puis redémarré.

FAQ

Pourquoi mon premier scan n'affiche aucune dérive ?
C'est le comportement attendu : le premier scan devient la baseline, il n'y a rien à quoi le comparer.
Peut-on surveiller un fichier unique (pas un dossier) ?
Oui — le chemin du target peut être un fichier ou un répertoire.
Quelle différence entre acquitter et promouvoir baseline ?
Acquitter : marque la dérive comme "vue". La baseline reste intacte, donc un futur scan retrouvera la même dérive. Promouvoir : adopte le scan courant comme nouvelle référence, la dérive disparaît du point de vue des scans suivants.
Comment supprimer les vieux scans ?
Soit activer BC_VIGIL_SCAN_RETENTION_DAYS pour purge automatique, soit supprimer un target (cascade tous ses scans), soit reset DB complet.
Quel algorithme choisir ?
Pour la détection de modifications malveillantes : sha256. Pour la détection de corruption accidentelle sur gros volumes : xxh128 (beaucoup plus rapide). crc32 : uniquement pour des cas légers, collisions plus probables.
Un schedule affiche "stuck", que faire ?
Vérifier que BitCrafts Vigil tourne sans interruption, regarder les scans récents du target pour voir s'il y a des échecs successifs ou des scans bloqués. Un redémarrage réinitialisera les jobs.
Peut-on consommer les événements d'un scan depuis un script ?
Oui, GET /scans/<id>/events.csv retourne un CSV (Content-Type text/csv), utilisable en curl + jq/csvkit ou dans un SIEM.
